# Acceptance Criteria — flow 061 (OpenTUI shell Phase 2 chrome parity)

- AC1: In the OpenTUI agent shell, assistant text renders as markdown (ATX headings, **bold**, `inline code`, fenced blocks, -/* bullets) via a WORKER-FREE converter (`markdownToChunks` → OpenTUI `StyledText` → `TextRenderable`) — the native `MarkdownRenderable` is avoided because its parser spins a WASM worker unavailable headless. Raw markers (`**`, leading `- `) are not shown; a bullet renders as `•`.
- AC2: Chrome parity blocks render with styling: a `● keryx` role header (accent + bold) + the user's `❯ line`; tool calls as `⚙ name(args)` (via the pure `summarizeToolArgs`); tool results collapsed as `↳ <first line> · +N more` (via `collapseToolOutput`), errors as `✗ …`; a dim `⋯ thinking` reasoning block; a dim `↑in ↓out tokens` usage line; system/error lines dim/red. Gutter = the transcript box padding.
- AC3: The reused logic stays in the pure helpers (`summarizeToolArgs`, `collapseToolOutput`) and the new pure `markdownToChunks`; `runAgentTurn`, the readline shell, chat mode, and `roleLabel` are unchanged. `--tui` opt-in + readline fallback (flow 060) are preserved.
- AC4: A headless test drives a scripted provider (with markdown + a tool call) through `runAgentTurn` + the TuiShell IO against `createTestRenderer` and asserts the captured frame contains the rendered assistant text (markers stripped, bullet glyph present) and the tool line. `bunx tsc --noEmit` clean; `bun test` green with no reduction from the flow-060 baseline (1496). No new dependency.
