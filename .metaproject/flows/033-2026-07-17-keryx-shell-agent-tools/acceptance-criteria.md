## Criteria

- AC1: Read-only builtin tools (interactive tool layer) — the durable `ToolExecutorPort.invoke` returns a hashed `ToolResult` receipt (`outputHash`, not content), so it cannot feed content back to a live model; Flow A therefore adds a lightweight, content-returning interactive tool layer that REUSES the neutral `NormalizedToolDefinition` shape (name + description + JSON `inputSchema` + `risk`) and the `ToolRisk` classes. `src/harness/tool/builtin/` defines three tools, all risk `read`: `get_cwd` (no input → the current working directory), `list_dir` (input `{path?}` → directory entries), and `read_file` (input `{path}` → file contents), each with a valid input JSON Schema. A real content-returning executor implements them against the filesystem, CONFINED to the project root: any `path` resolving outside the root (via `..` or an absolute escape) yields an error result (`{ isError: true }`) and NEVER reads outside the root. Unit tests cover a happy read, a `..`/absolute path-escape rejection, and a missing-file error — deterministic, no network.
- AC2: Interactive agent driver — a new driver (e.g. `src/commands/agent.ts` `runAgentTurn`) builds a `NormalizedRequest` carrying `tools` (the builtin definitions) and a trusted system instruction, streams it, renders assistant `text_delta`s, and on each `tool_call_end`: validates the tool `input` against its `inputSchema` (reusing `validateAgainstSchemaObject`) and applies a risk gate (Flow A: `read` → allow; any non-`read` risk → denied, since only read tools exist here); on allow it invokes the content-returning executor, appends the tool output as a `role:"tool"` message, and re-requests the provider — looping until a text-only finish or a `maxToolCalls` guard, then returning the final assistant message. The driver is deterministic (injected provider/tools/executor/clock/idSeq; no `Date.now`/`Math.random`) and offline-testable; a unit test drives it with a FakeProvider transcript that emits a `get_cwd`/`list_dir` tool call and asserts the executor ran and its output was fed back into the next request's `messages` (role `tool`). (The full durable `ToolExecutorPort` + policy-engine `decide` + receipts integration is a later SA-01 flow, alongside `shell_exec`/approval.)
- AC3: Metaproject context injection — the driver's trusted `systemInstruction` is assembled by a pure builder that embeds a compact `keryx orient` orientation block when available and falls back to a minimal static instruction when it is not; the assembled instruction has provenance `trusted` (never `project`/`model`). Unit-tested for BOTH the orient-present and orient-absent paths (the builder never throws when orient output is missing/empty).
- AC4: Opt-in, non-breaking wiring — `keryx shell --agent` selects the agent driver (a mid-session `/agent` toggle is deferred to a later SA-01 flow); without the flag the existing deterministic chat `runShell` core is used UNCHANGED. The flow-021/022/031/032 `runShell` tests remain green with zero modifications to the chat core's behavior. Agent-mode rendering reuses `createRichIo` with additive `onToolCall`/`onToolResult` hooks (a tool call and its result are shown, styled, without corrupting the stream or the status bar); the real content-returning executor and the `read`-allow risk gate are injected in the (not-unit-tested) TTY wrapper. No new production dependency — `dependencies` REMAINS `{}`. `shell_exec`, approval UX, metaproject tools, and the token counter are NOT introduced (later SA-01 flows).
- AC5: No regression / offline / deterministic + live proof — `tsc --noEmit` is clean and the full `bun test` suite is ≥ the pre-change baseline of 1369 pass / 3 skip / 0 fail with the new tool/driver/context tests green and 0 fail; the entire automated suite runs OFFLINE and deterministic (FakeProvider + injected fs root; no real TTY or network). A live manual smoke (`bun src/cli.ts shell --agent` against a tool-capable model) shows the agent answering "what is my current directory / list the files here" by actually CALLING `get_cwd`/`list_dir` and reporting the REAL cwd/entries (no hallucinated path) — recorded in the flow journal; not a CI gate. The frozen requirements package, canonical schemas, ADR-0001…0004, `runOffline`, and the `runShell` chat core are NOT modified.
