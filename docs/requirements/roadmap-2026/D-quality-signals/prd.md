## Executive Summary

gd-metapro's `health` module scores code quality by combining multiple tools (eslint,
typescript, tests, coverage, dependency-audit, complexity) into a normalized 0–100 score
and a `pass/warn/fail` gate consumed by `flow`. It already *collects* per-file git churn
(`getChurn`) and per-function cyclomatic complexity (`source-analysis.ts`), but it never
combines them into the single highest-signal behavioral metric the industry (CodeScene)
relies on: **hotspots — files that are both complex and frequently changing.** Block D's
**D1** turns those two already-available inputs into an additive hotspot signal that feeds
the existing score and gate, purely and deterministically, with no new dependency.

gd-metapro's `testing` module selects which tests to run for a change using a **static**
changed-file heuristic (test-file matches + naming/directory proximity). This over- and
under-selects: it can miss a test in a distant directory that genuinely covers the changed
code, and run tests that don't. Block D's **D2** adds an opt-in **coverage map**
(test → covered source files/lines, parsed from coverage tooling the repo already produces)
so selection matches what tests *actually* cover, plus an **always-on smoke tier (D3)** that
runs on every invocation to catch cross-cutting regressions. When no map exists, selection
falls back **byte-identically** to today's static heuristic.

Block D is **who**: the *agent operator* running gd-metapro in their repo (sharper health
signals, sharper test selection) and the `flow` gate that consumes health. It is **deterministic
and local end-to-end** — no model, no network, no new runtime dependency.

## User Stories

### Epic 1 — D1: Git-churn × complexity hotspot signal (health)

#### US-D101: See which files are both complex and churning
**As a** agent operator
**I want to** get a ranked list of hotspot files (high churn × high complexity)
**So that** I can focus review/refactor effort where risk actually concentrates (goal G-D1, problem P-D1).

**Acceptance Criteria:**
- [ ] `gd-metapro health run` emits a deterministic **hotspot ranking**: for each source file, `hotspot = churn(file) × complexity(file)`, sorted descending with a stable path tiebreak.
- [ ] On the `churn/complexity` fixture (seeded git history + known complexity), the top-N ranked files equal the expected hotspot set exactly.
- [ ] Re-running the generator twice on the fixture yields an identical ranking (reproducible, `XP4`).
- [ ] Churn reuses the existing `getChurn(cwd, churnWindowDays)`; complexity reuses `analyzeSourceFiles` — **no new dependency and no network call** are introduced.

**Technical Notes:**
- Constraint ref: `D-1` (pure, dep-free, additive), `F-2`/`F-4` (measured against a committed fixture), `XP4` (reproducible).
- Architecture ref: §5 Group D row 1 (git-churn × complexity, additive to score/gate); §6 `churn/complexity` corpus.

**Priority:** P0

#### US-D102: Feed the hotspot signal into the health score and gate without breaking existing scores
**As a** `flow` health gate
**I want to** optionally weight hotspots into the normalized health score
**So that** high-risk hotspots can escalate the gate — while the default behavior is unchanged (goal G-D1, NG-D2).

**Acceptance Criteria:**
- [ ] The hotspot signal enters `healthScore()` **additively** through the same per-LOC normalization as the existing risk/coverage/complexity penalties (a new `hotspotPenalty`).
- [ ] `hotspotPenalty` is governed by a config weight `scoring.hotspotWeight` that **defaults to 0**, so with default config every scope's `health_score` and the gate `status` are **identical to the pre-D1 values** (regression_score == 0 on the fixture) — the "added without breaking existing score" guarantee (`D-1`, `NG-D2`, `C0-7`).
- [ ] Setting `scoring.hotspotWeight > 0` measurably lowers the score for scopes containing fixture hotspots and can escalate the gate via the existing regression path — **no new gate rule** is added, the gate contract is preserved (`D-4`).
- [ ] The added report fields are additive/nullable and versioned via a `schemaVersion` bump; the numeric score/gate decisions are unchanged at default config.

**Technical Notes:**
- Constraint ref: `D-1`, `D-4`, `NG-D2`, `C0-7`, `C0-8` (config deep-merge over defaults).
- Architecture ref: §5 Group D (additive to the existing score/gate); existing `src/health/scoring.ts:healthScore` + `src/health/gate.ts:computeGate`.

**Priority:** P0

### Epic 2 — D2: Dynamic coverage-map Test Impact Analysis (testing)

#### US-D201: Build/import a test-to-code coverage map
**As a** agent operator
**I want to** produce a coverage map (which tests cover which source files/lines) from the coverage output my runner already emits
**So that** test selection can be based on real coverage, not naming guesses (goal G-D2, problem P-D2).

**Acceptance Criteria:**
- [ ] A `gd-metapro testing coverage-map build` subcommand runs the suite with coverage (or imports an existing coverage report) and writes a deterministic `coverage-map.json` mapping `testFile → { coveredFiles[], coveredLines? }`.
- [ ] The map is parsed from **existing** coverage tooling output (lcov and/or V8/bun JSON) — **no new dependency and no bespoke instrumentation** on the default path (`D-6`, `NG-D3`, `NG-D4`).
- [ ] The map is deterministic and git-diffable (stable key order, sorted arrays); re-building on the fixture yields an identical file (`XP4`, `F-2`).
- [ ] The coverage-map capability wires all four coordinated parts (`C0-3`): an `init` flag (`--testing-tia` / `--no-testing-tia`), a `metaproject.json` manifest capability entry, a `testing.config.json` toggle deep-merged over defaults, and a fallback contract.

**Technical Notes:**
- Constraint ref: `D-2`, `D-6`, `C0-3`, `C0-8`, `C0-9`, `NG-D3`, `NG-D4`.
- Architecture ref: §5 Group D row 2 (parse existing coverage output → change→impacted-test map, opt-in); §8 mapping (testing service + config).

**Priority:** P0

#### US-D202: Select the tests that actually cover the changed lines
**As a** agent operator
**I want to** `gd-metapro testing run --changed` to select tests whose coverage intersects the changed files/lines
**So that** the selected set is more precise than the static naming heuristic (goal G-D2, problem P-D2).

**Acceptance Criteria:**
- [ ] When a coverage map is present, changed-scope selection = the set of tests whose `coveredFiles` (or `coveredLines`, when line info exists) intersect the changed files/lines from `getChangedFiles`.
- [ ] On the `change→impacted-test` fixture (known change → true impacted-test mapping + a coverage map), coverage-map selection has **higher precision** than the static changed-file selection.
- [ ] Files not present in the map (e.g. brand-new files) still fall back to the existing naming/directory heuristic, composed (unioned) with the map-based set — no changed file is silently dropped.
- [ ] The report's `selection.strategies` reflects `"coverage-map"` when the map drove selection (observable/testable).

**Technical Notes:**
- Constraint ref: `D-2`, `F-1`, `F-4`, `T-3` (availability-true path test).
- Architecture ref: §5 Group D row 2; existing `src/testing/service.ts:selectChangedTests`.

**Priority:** P0

#### US-D203: Preserve static changed-file selection as the byte-identical default/fallback
**As a** agent operator with no coverage map
**I want to** `testing run --changed` to behave exactly as it does today
**So that** the zero-config deterministic default never regresses (goal G-D2, NG-D3, C0-7).

**Acceptance Criteria:**
- [ ] With the coverage-map capability disabled OR no `coverage-map.json` present, `selectChangedTests` returns a selection **byte-identical** to today's output on the same fixture (same `selectedTests`, `changedFiles`, `fallback`).
- [ ] No coverage-map file is read, no coverage command is run, and the report's `selection.strategies` remain today's `["runner","gdgraph","naming"]` when the map is absent.
- [ ] A test asserts the fallback selection equals the pre-D2 selection (the `XP2` fallback, `F-3`).

**Technical Notes:**
- Constraint ref: `D-2`, `C0-4`, `C0-5`, `C0-7`, `F-3`, `T-3` (availability-false fallback test).
- Architecture ref: §5 Group D ("falls back to today's static changed-file selection when no map"); §7 (opt-in fallback is a first-class code path).

**Priority:** P0

### Epic 3 — D3: Always-on smoke tier (testing)

#### US-D301: Run an always-on smoke tier alongside scoped selection
**As a** agent operator
**I want to** a configured smoke set to run on every `testing run` regardless of scope
**So that** cross-cutting regressions outside the changed scope are still caught (goal G-D3, problem P-D3).

**Acceptance Criteria:**
- [ ] A `testing.config.json` `smoke` block (selectors: globs / tags / explicit test list) defines the smoke set; when configured, the smoke tests are **unioned into every** invocation's selected set (changed, scope, or project).
- [ ] The smoke tier composes with (never suppresses) scoped selection: the final set = scoped-selection ∪ smoke-set; the report distinguishes the smoke contribution (observable/testable, `D-3`).
- [ ] With **no** `smoke` block configured (default), the selected set is byte-identical to today (empty smoke ⇒ no change, `C0-7`).
- [ ] An integration test asserts smoke inclusion across changed/scope/project selection modes.

**Technical Notes:**
- Constraint ref: `D-3`, `C0-7`, `T-3`, `T-4` (no-network sandbox: selection is pure/offline).
- Architecture ref: §5 Group D row 3 ("always-on smoke tier = pure orchestration composed with scoped selection").

**Priority:** P1

## Data Model (High-Level)
- **Health (D1)** — `ScopeMetrics` gains a nullable `hotspot` aggregate; the report gains a nullable `hotspots: Array<{ file, churn, complexity, score }>` ranking. `HealthConfig` gains `scoring.hotspotWeight` (default 0) and `metrics.hotspotThreshold`. `schemaVersion` bumps (additive fields only). No churn/complexity storage changes — both are already computed.
- **Testing (D2/D3)** — new derived artifact `data/testing/coverage-map.json` (`{ schemaVersion, generatedAt, map: { [testFile]: { coveredFiles: string[]; coveredLines?: Record<string, number[]> } } }`). `TestingConfig` gains a `coverageMap` toggle block and a `smoke` block. `TestingReport.selection` gains `smokeTests: string[]` and allows `"coverage-map"` in `strategies`.

## API Surface (High-Level)
- `health run` — unchanged signature; report additionally carries the hotspot ranking; score/gate unchanged at default weight.
- `testing coverage-map build` (new) / `testing coverage-map status` — build/import and inspect the map.
- `testing run --changed` — now coverage-map-aware when a map exists; static otherwise; always unions the smoke tier.
- No MCP/network surface in Block D (out of scope for D; MCP is Block A).

## Security Requirements
- Testing already routes raw runner output through `guardOutput`/`redactRaw` before persisting (leak-safe). The coverage-map builder MUST reuse the same write seam for any raw coverage log it persists, so `XP4`/leak-safety is preserved (no raw secret/PII in committable artifacts). The `coverage-map.json` itself contains only file paths and line numbers — no runner stdout.

## Testing Requirements
_Aligned with `tech-bestpractices.md` §7–§8 and the test-pyramid target._
- **Unit (~70%)**: `hotspotScore`/`hotspotPenalty` purity + ranking; coverage-map parser (lcov + V8/bun JSON); coverage-map selection intersection; static-fallback equality; smoke-union composition.
- **Integration (~25%)**: `health run` end-to-end score-invariance at default weight; `testing run --changed` map-driven vs static; smoke inclusion across scopes; config/manifest capability wiring (`C0-3`).
- **E2E/CI (~5%)**: fixture-corpora acceptance (`churn/complexity`, `change→impacted-test`) via Block 0's harness; no-network sandbox (`T-4`) confirming selection and hotspot computation open no socket.
- **Fixtures (`F-1`)**: `fixtures/churn-complexity/` and `fixtures/change-impacted-test/`, deterministic and git-committed; each named as the acceptance gate in `acceptance-criteria.md`.

## Out of Scope
Everything in NG-D1..NG-D4 above, plus: MCP exposure of health/testing (Block A), any model/embedding backend, any hosted service, and re-speccing the shipped `health`/`testing` cores.

## Open Questions
- **OQ-1** `hotspotWeight` default is 0 to guarantee byte-identical scores. Should the block ALSO ship a recommended non-zero preset (documented, opt-in) so operators get value without hand-tuning? (Proposed: yes, documented in `specification.md`, still default 0.)
- **OQ-2** Adding nullable report fields requires a `schemaVersion` bump; existing golden-JSON tests must be updated for the additive schema (not for changed scores). Confirm the consistency-checker treats an additive-schema bump as compatible with `C0-7` (behavior byte-identical; schema additive).
- **OQ-3** Coverage-line granularity (`coveredLines`) depends on the runner emitting line data (lcov `DA:` / V8). When only file-level coverage exists, selection degrades to file-level intersection (still more precise than naming). Confirm file-level-only is acceptable for the fixture's precision target.
