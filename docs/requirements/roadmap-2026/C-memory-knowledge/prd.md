## Executive Summary

gd-metapro's knowledge layer (`memory` + `gdwiki`) stores git-diffable Markdown and
retrieves it with a deterministic, weighted **lexical** search. That is auditable and
zero-dependency, but it has a **paraphrase/semantic recall ceiling**, no way to say a
decision was **superseded** (facts are overwritten by dedup, so "what did we believe
when" is unanswerable), no **type** distinction that would let the right memory be
**injected** into an implementer's prompt, and no **programmatic Q&A** surface for
external agents.

Block C closes these four gaps as **opt-in ceilings** on the deterministic floor. C1 adds
a **local embedding index** as a derived, disposable adapter that improves recall while
lexical search stays the default and fallback. C2 adds **bitemporal validity fields** to
Markdown entries so decisions can be superseded (event-time vs ingestion-time) rather than
dedup-overwritten. C3 maps memory kinds to **semantic / episodic / procedural** classes
and **actively injects** procedural memory into task-implementer prompts. C4 exposes wiki
+ memory as **MCP Resources** and a `wiki ask` action riding Block A's stdio surface.

The Markdown store stays authoritative throughout: every derived layer is rebuildable
from, or expressed inside, the committed Markdown, and with nothing enabled the modules
behave byte-identically to today.

## Non-Goals (from problem-statement.md — MUST NOT appear as features)

- **NG-C1** — No hosted/cloud embedding API, no external vector-DB service. Local only.
- **NG-C2** — Do **not** replace the deterministic Markdown store; the index is derived,
  disposable, and non-authoritative.
- **NG-C3** — No graph-database / Zep-Graphiti-style memory backend; temporal model is
  file-native.
- **NG-C4** — Not a general chat/RAG assistant over arbitrary corpora; scope is the
  metaproject's own `memory` + `wiki` data.

## User Stories

### Epic C1 — Optional embedding index for `memory` search

#### US-C101: Deterministic lexical search stays the default and the fallback
**As an** agent operator
**I want** `memory search` to run its current weighted-lexical scoring by default
**So that** enabling nothing keeps retrieval auditable, zero-dep, and byte-identical (G-C1)

**Acceptance Criteria:**
- [ ] With no `index` capability configured, `memory search` output (`latest.md`,
      `latest.json`, ordering, scores) is byte-identical to today's on a fixed corpus.
- [ ] No embedding runtime import occurs and no asset is touched on the default path.
- [ ] The `search.ts` weighted scoring (`relevance/recency/confidence/status/scope`)
      is unchanged when the capability is off.

**Technical Notes:** Constraint ref: C-2, C0-4, C0-7, XP2. Architecture ref: §5 Group C.
**Priority:** P0

#### US-C102: Opt-in local embedding index improves paraphrase recall
**As an** agent operator
**I want** an opt-in local embedding index layered above the Markdown store
**So that** paraphrased / semantic queries recall the right memory the lexical path misses (G-C1)

**Acceptance Criteria:**
- [ ] Enabling `memory.embedding` via the four coordinated parts (init flag, manifest
      entry, `memory.config.json` toggle, `resolveCapability`) activates an embedding
      rerank/augment path over the lexical candidate set.
- [ ] On the committed **paraphrase** fixture (query→expected-memory incl. paraphrases),
      recall@k with the index is **measurably higher** than lexical-only.
- [ ] The index is a **derived, disposable** artifact under `.metaproject/data/memory/`;
      deleting it and re-running rebuilds it; the Markdown store is never mutated by
      indexing.
- [ ] The embedding runtime is an `optionalDependency` lazy-loaded only inside
      `memory/embedding/adapter.ts`; the model is an XP3 asset resolved via the Asset
      Resolver.

**Technical Notes:** Constraint ref: C-1, C-3, A-1..A-3, C0-2, C0-3, F-1, F-4.
Architecture ref: §2, §3, §5 Group C. **Priority:** P0

#### US-C103: Absent/failed backend degrades gracefully to lexical
**As an** agent operator
**I want** `memory search` to warn-once and fall back when the backend/model is missing
**So that** an enabled-but-unavailable index never hard-fails a command (G-C1)

**Acceptance Criteria:**
- [ ] Capability enabled but runtime not installed / model absent / checksum fails ⇒ warn
      once to stderr, run lexical search, **exit 0**.
- [ ] Adapter throwing at runtime is caught and degrades to the lexical result.

**Technical Notes:** Constraint ref: C0-5, C0-11, A-3, T-3. Architecture ref: §2.3.
**Priority:** P0

### Epic C2 — Bitemporal fact model

#### US-C201: Decisions can be superseded with validity intervals
**As an** agent operator
**I want** memory entries to carry `Valid-From` / `Valid-To` / `Superseded-By` fields
**So that** a later decision supersedes an earlier one instead of dedup-overwriting it (G-C2)

**Acceptance Criteria:**
- [ ] Memory entry header supports `Valid-From`, `Valid-To`, `Superseded-By`,
      `Recorded-At` (ingestion-time) fields, parsed by `store.ts:parseEntry`.
- [ ] Superseding an entry sets the old entry's `Valid-To` + `Superseded-By` and status
      `superseded`, and links the new entry's `Supersedes` — **both entries remain on
      disk** (no destructive overwrite).
- [ ] Fields are plain Markdown, git-diffable, and reproducible; no database is introduced.

**Technical Notes:** Constraint ref: C-4, NG-C3, XP4. Architecture ref: §5 Group C.
**Priority:** P0

#### US-C202: "Current" vs "as-of" queries resolve correctly
**As an** agent operator
**I want** `memory search --as-of <date>` and default "current" filtering
**So that** I can ask "what did we believe when" and never see a superseded fact as current (G-C2)

**Acceptance Criteria:**
- [ ] Default `memory search` excludes entries whose `Valid-To` is in the past or that are
      `Superseded-By` another (event-time filter).
- [ ] `--as-of <date>` returns the entry whose validity interval contains that date.
- [ ] On the committed **temporal** fixture (supersession chains + as-of queries),
      resolution is 100% correct.

**Technical Notes:** Constraint ref: C-4, F-1, F-4. Architecture ref: §5, §6.
**Priority:** P0

### Epic C3 — Memory typing + procedural injection

#### US-C301: Every memory kind maps to a semantic/episodic/procedural class
**As an** agent operator
**I want** each `MEMORY_TYPES` kind mapped to a `semantic` / `episodic` / `procedural` class
**So that** memory can be retrieved by role, not only by text match (G-C3)

**Acceptance Criteria:**
- [ ] A committed type-mapping table maps all 11 existing kinds (lesson, decision,
      constraint, known-mistake, pattern, …) to exactly one class.
- [ ] The class is exposed as a memory header/metadata field and parsed into `MemoryEntry`.
- [ ] A type-scoped query (`--class procedural`) returns **only** procedural memories.

**Technical Notes:** Constraint ref: C-5. Architecture ref: §5 Group C. **Priority:** P0

#### US-C302: Procedural memory is injected into task-implementer prompts
**As an** implementer agent
**I want** relevant **procedural** memory injected into my prompt at flow time
**So that** I follow known procedures/constraints without having to search for them (G-C3)

**Acceptance Criteria:**
- [ ] A `proceduralMemoryForScope(cwd, scope)` retrieval path (generalizing the existing
      `relevantAcceptedMemory` seam) returns accepted procedural entries for a task scope.
- [ ] The flow / task-implementer prompt assembly injects the rendered procedural block;
      a flow-task integration test asserts the block is present in the assembled prompt.
- [ ] With no procedural memories in scope, injection is an empty no-op (prompt unchanged).

**Technical Notes:** Constraint ref: C-5; existing seam `src/memory/relevant.ts` +
`src/gdskills/verify.ts`. Architecture ref: §8 (C feeds flow). **Priority:** P0

### Epic C4 — gdwiki Q&A / MCP endpoint

#### US-C401: Wiki + memory exposed as read-only MCP Resources
**As an** external coding agent
**I want** `wiki` and `memory` exposed as MCP Resources over stdio
**So that** I can list and read the knowledge base programmatically (G-C4)

**Acceptance Criteria:**
- [ ] `resources/list` + `resources/read` expose `wiki` pages and `memory` entries
      read-only through Block A's `src/mcp/resources.ts`.
- [ ] All resource output passes through `security/guard.ts:redactRaw` as untrusted.
- [ ] `modules.mcp.enabled=false` ⇒ nothing exposed; deterministic collect path unchanged.

**Technical Notes:** Constraint ref: C-6, M-1, M-4, M-5, M-7, XP5. Architecture ref: §4.
**Priority:** P1

#### US-C402: `wiki ask` answers a question via deterministic retrieval (+ optional rerank)
**As an** external coding agent
**I want** a `wiki ask "<question>"` action (MCP Tool + CLI) with deterministic retrieval
**So that** I get a cited answer assembled from wiki/memory pages, no hosted service (G-C4)

**Acceptance Criteria:**
- [ ] `wiki ask` returns ranked page/entry excerpts with source paths via **deterministic
      lexical retrieval** by default; when C1's embedding capability is on, an optional
      rerank improves ordering.
- [ ] Exposed as an MCP Tool that is a thin adapter over the wiki/memory service facades
      (no new business logic in `src/mcp/`).
- [ ] An MCP round-trip (`tools/list` → `tools/call wiki.ask`) resolves a question against
      a fixture project over stdio in CI.
- [ ] With the endpoint disabled, `gdwiki collect` output is byte-identical to today.

**Technical Notes:** Constraint ref: C-6, C-8, M-2, M-3, M-6, T-2. Architecture ref: §4, §8.
**Priority:** P1

## Data Model (high-level)

- **Memory entry (extended)** — existing `MemoryEntry` (`store.ts`/`types.ts`) gains:
  `class` (`semantic|episodic|procedural`), `validFrom`, `validTo`, `recordedAt`,
  `supersededBy`, `supersedes`. All parsed from Markdown header/section fields; all
  optional with today's entries defaulting to `class` by type-map and open-ended validity.
- **Embedding index** — `.metaproject/data/memory/embeddings/` : derived, disposable
  `{entryPath, vector, contentHash, model, generatedAt}` records + an `index.meta.json`.
  Never authoritative; rebuildable from Markdown.
- **Type-map** — `MEMORY_CLASS_MAP: Record<MemoryType, MemoryClass>` constant, committed.

See `specification.md §Schema` for exact field names and the frontmatter grammar.

## API Surface (high-level)

- CLI: `memory search [--as-of <date>] [--class <c>] [--k <n>]`,
  `memory supersede <old> --by <new>`, `memory index [--embeddings]`,
  `memory assets pull <id>` (via Block 0), `wiki ask "<q>"`.
- Service: `MemoryService.search` (extended filters), new `MemoryService.supersede`,
  new `proceduralMemoryForScope`; `GdWikiService.ask`.
- MCP: Resources `wiki`, `memory`; Tool `wiki.ask` (and `memory.search`). [M-2]

## Security Requirements

- MCP resource/tool output routed through `redactRaw` as untrusted. [M-5]
- No raw secret/PII written into the embedding index or any derived artifact; leak-safety
  preserved. [E-9-inherited, XP4]
