## Executive Summary

gd-metapro's generated intelligence layer (graph, security, flow, memory, health,
wiki) is today reachable **only** through `gd-metapro <module> <cmd>` CLI calls
described in Markdown — a closed, gd-metapro-only loop. Every external coding agent
in the target set speaks **MCP**. Block A closes that gap with a thin, stdio-first
MCP protocol adapter (`src/mcp/`) that exposes the *same in-process service
contracts the CLI uses* as MCP **Tools**, and the *already-generated on-disk
artifacts* as read-only MCP **Resources** — adding no new module logic.

Because the new MCP surface is itself a new, unguarded trust boundary, **E3 ships
with A**: all tool output is routed through the existing `redactRaw` seam as
untrusted from the first commit, and a new deterministic `security scan-mcp`
detector flags tool-poisoning / line-jumping / rug-pull in MCP tool manifests.

Block A also makes gd-metapro's outputs **portable**: an emitted `llms.txt`
discovery file, a gdskills plugin/marketplace export, and a doc-only repositioning
of the "Metaproject Standard" from a rival whole-workspace standard into a
**generator** of the three LF-consolidated artifacts (AGENTS.md + Agent Skills +
MCP server). All of this is opt-in: with `modules.mcp.enabled=false` the
deterministic core is byte-identical to today and no SDK loads.

### Package-wide gate (inherited, binds A)
- With `modules.mcp.enabled=false` and no SDK installed, the full existing suite and
  every deterministic command behave **byte-identically** to today (C0-7).

## Non-Goals

- **NG-A1**: No hosted/multi-tenant MCP service; **no default HTTP/SSE listener**.
  Transport is stdio-first (XP5); HTTP is a *separate* opt-in behind `--http`.
- **NG-A2**: Not authoring/ratifying a new public interop standard — we *emit*
  existing LF standards, we do not compete with them.
- **NG-A3**: No auth/authorization/identity/multi-tenant layer for MCP access.
- **NG-A4**: No write/mutating tool that bypasses a deterministic gate (e.g. no MCP
  tool that force-completes a `flow` without its AC/PR/health gate).

## Technical Foundation

- **Stack (inherited, not re-chosen)**: Bun + TypeScript, ESM, layered
  `cli → commands → services → lib`; zero-dep / zero-network deterministic core.
- **Architecture (from `architecture.md` §4)**: new cross-cutting package `src/mcp/`
  (peer of `standard/`, `security/`); Tools = thin adapters over `createXService()`
  methods; Resources = read-only on-disk artifacts; discovery is manifest-driven from
  `metaproject.json`; E3 reuses the `security/detect/` + `guard.redactRaw` seams.
- **Key MUST constraints (from `tech-bestpractices.md` §0/§2)**:
  - C0-1 `dependencies` stays empty; MCP SDK is an `optionalDependency` (M-6).
  - C0-2 no top-level import of the SDK — lazy `await import()` inside the server only.
  - M-1 stdio default, no listening socket on the default path; M-8 no default HTTP.
  - M-2/M-3 Tools are thin; `src/mcp/` imports only service facades + `lib` + `guard`.
  - M-4 Resources read-only; M-5 **all** tool output through `redactRaw` (E3).
  - M-7 `modules.mcp.enabled=false` ⇒ unchanged, no SDK loaded.
  - M-10 no mutating tool bypasses a deterministic gate.
  - **Sanctioned exception**: `mcp serve` MAY hard-require its SDK (user explicitly
    invoked a server) — the only opt-in command allowed to hard-fail on a missing dep.

## User Stories

### Epic 1 — MCP Tool surface (G-A1)

#### US-A101: List and invoke module Tools over stdio
**As an** external coding agent (Cursor/Claude Code/Codex/Devin)
**I want to** discover and invoke gd-metapro module actions via MCP `tools/list` and
`tools/call` over stdio
**So that** I can consume the workspace's intelligence layer without gd-metapro-specific glue (solves P-A1, goal G-A1).

**Acceptance Criteria:**
- [ ] `gd-metapro mcp serve` starts a stdio JSON-RPC MCP server that responds to the
  MCP initialize handshake.
- [ ] `tools/list` returns ≥ the review-named actions: `gdgraph.affected`,
  `gdgraph.cycles`, `gdgraph.orphans`, `security.check`, `security.scan`,
  `flow.status`, `memory.search`, `health.gate`, `wiki.query`, `standard.validate`.
- [ ] `tools/call` on each returns the same typed result the corresponding
  `createXService()` method returns (JSON-serialized), verified against an
  in-process unit test of that method (T-1).
- [ ] No new module business logic lives in `src/mcp/` — each Tool handler only
  (de)serializes params and calls one service method.

**Technical Notes:**
- Constraint ref: M-1, M-2, M-3, M-6, T-1, T-2.
- Architecture ref: §4.1 (Tools = thin adapters over `createXService()`), §8 (Block A row).

**Priority:** P0

#### US-A102: Expose generated artifacts as read-only Resources
**As an** external coding agent
**I want to** browse and read `data/*/artifacts`, `wiki`, and `memory` via MCP
`resources/list` and `resources/read`
**So that** I can inject gd-metapro's generated context without shell access (G-A1).

**Acceptance Criteria:**
- [ ] `resources/list` enumerates ≥3 Resource classes: `data/*/artifacts`, `wiki`, `memory`.
- [ ] `resources/read` returns the raw file contents for a listed Resource URI.
- [ ] Resources are strictly read-only — no MCP call mutates any file.
- [ ] Resource URIs follow the documented scheme (`specification.md` §Resource-URI).

**Technical Notes:**
- Constraint ref: M-4, T-2.
- Architecture ref: §4.1 (Resources = read-only generated artifacts).

**Priority:** P0

#### US-A103: Manifest-driven discovery
**As an** agent operator
**I want** the set of exposed Tools/Resources to be derived from `metaproject.json`
**So that** disabled modules never appear as MCP surface and discovery stays consistent (G-A1).

**Acceptance Criteria:**
- [ ] A module disabled in `metaproject.json` (`modules.<m>.enabled=false`) does not
  appear in `tools/list` or `resources/list`.
- [ ] `modules.mcp.enabled=false` ⇒ server refuses to expose anything / the command
  behaves as unchanged (M-7); no SDK loaded on any other path.
- [ ] Tool/Resource registration is derivable from the manifest (M-11).

**Technical Notes:**
- Constraint ref: M-7, M-11, C0-9.
- Architecture ref: §4.1 (opt-in module in `metaproject.json`).

**Priority:** P0

#### US-A104: HTTP/SSE as a separate opt-in
**As an** agent operator whose client needs HTTP
**I want** `gd-metapro mcp serve --http` behind its own capability entry
**So that** the offline stdio floor stays the default and HTTP is fully removable (G-A1, NG-A1).

**Acceptance Criteria:**
- [ ] Default `mcp serve` opens **no** listening socket (M-1).
- [ ] `--http` requires its own capability toggle; without it, no HTTP path is reachable.
- [ ] stdio and HTTP transports are isolated in `transport/` so HTTP is removable (M-12).

**Technical Notes:**
- Constraint ref: M-1, M-8, M-12, XP5.
- Architecture ref: §4.2 (transport).

**Priority:** P1

### Epic 2 — E3: MCP-boundary security (ships with A)

#### US-A201: Route all MCP tool output through `redactRaw`
**As an** agent operator
**I want** every MCP tool result routed through `security/guard.ts:redactRaw` as
untrusted before it leaves the server
**So that** the new MCP boundary cannot leak secrets/PII from day one (P-E3, E3).

**Acceptance Criteria:**
- [ ] Every Tool handler passes its serialized output through `redactRaw({ cwd,
  content, source: "tool-output" })` before returning it over the transport.
- [ ] With `security` disabled, `redactRaw` returns byte-identical content (existing
  seam contract) — no behavior change, no crash.
- [ ] A test asserts a seeded secret in a tool result is masked in the transported output.

**Technical Notes:**
- Constraint ref: M-5, E-5, C0-11 (never throws).
- Architecture ref: §4.3 (reuse `redactRaw`; ships with A1 from first commit).

**Priority:** P0

#### US-A202: `security scan-mcp` detector
**As an** agent operator adopting MCP
**I want** `gd-metapro security scan-mcp` to statically analyze MCP tool manifests
for tool-poisoning / line-jumping / rug-pull
**So that** a poisoned third-party (or local) MCP tool is caught before use (P-E3, E3).

**Acceptance Criteria:**
- [ ] `security scan-mcp <manifest|dir>` flags every vector in the committed
  `fixtures/mcp-threat/` corpus (poisoning, line-jumping, rug-pull).
- [ ] The detector is pure and network-free, implemented as `security/detect/mcp.ts`
  returning `DetectorMatch[]`, slotting into `runDetectors` (E-3).
- [ ] Findings are leak-safe (masked; no raw manifest secret in artifacts) (E-9).

**Technical Notes:**
- Constraint ref: E-3, E-5, E-9, F-1, F-2, F-4.
- Architecture ref: §4.3, §6 (`mcp-threat` fixture corpus is the acceptance gate).

**Priority:** P0

## Data / Surface Model (High-Level)

- **Tool registry** (`src/mcp/tools.ts`): a static array of
  `{ name, module, method, inputSchema, outputShape, mutating }` entries, each
  mapping one MCP Tool to exactly one `createXService()` method (see
  `specification.md` §Tool-Registry). No entry may be `mutating` unless it calls a
  gate-preserving service method (M-10).
- **Resource registry** (`src/mcp/resources.ts`): read-only mapping from a URI
  scheme to on-disk paths under `.metaproject/data/*/artifacts`, `wiki`, `memory`.
- **`scan-mcp` detector** (`src/security/detect/mcp.ts`): pure function
  `(manifest) → DetectorMatch[]`; slots into `runDetectors`.

## Security Requirements

- All tool output routed through `redactRaw` as untrusted (M-5, E3).
- `scan-mcp` findings are leak-safe (masks/HMAC; no raw secret in artifacts) (E-9).
- No auth layer (NG-A3); stdio is offline-safe by default (M-1).
- No mutating tool bypasses a deterministic gate (M-10, NG-A4).

## Testing Requirements

- In-process unit test per service method exposed as a Tool, independent of MCP
  transport (T-1).
- stdio `tools/list`+`tools/call` and `resources/list`+`resources/read` round-trip
  test in CI against a fixture project (T-2).
- Availability-true + availability-false tests for the MCP capability (T-3): SDK
  present ⇒ server runs; SDK absent ⇒ default paths unchanged, `mcp serve` emits an
  actionable error (sanctioned exception).
- No-network sandbox test: every default command succeeds with no socket opened (T-4).
- `mcp-threat` fixture corpus committed; `scan-mcp` measured against it, not prose (F-1..F-4).

## Out of Scope

Everything in NG-A1..NG-A4. Also: Group C4's wiki *semantic* Q&A beyond the
deterministic wiki Resource/Tool (that rides A's surface but is specified in Block C).
