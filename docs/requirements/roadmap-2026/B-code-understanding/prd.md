## Summary

Today `gdgraph build` walks the source tree, extracts import specifiers with
`Bun.Transpiler.scanImports` (falling back to four import regexes in
`extractImportSpecifiersFallback`), and emits a **file-level** graph: nodes of
kind `file`/`asset`, edges of kind `imports`/`asset`/`unresolved`, into
`.metaproject/data/gdgraph/storage/{nodes,edges}.jsonl` plus
`artifacts/{module-map.json,summary.md}`. `affected` reports **one hop** of
dependents/dependencies; there is **no repo map**.

This block adds three capabilities on top of that floor, all instantiating the
Block 0 seam:

1. **B1** — an opt-in `web-tree-sitter` (WASM) parser that resolves
   **function / class / method** symbols and **CALL** edges, written to *new,
   additive* storage files so the legacy file-level artifacts stay byte-identical.
   When `web-tree-sitter` or its grammars are absent, `gdgraph` runs exactly
   today's regex/scan path (`B-1`, `C0-7`).
2. **B2** — `affected --depth N`: a pure BFS/DFS transitive closure over
   dependents with a ranked blast-radius. `--depth 1` reproduces today's output.
3. **B3** — `gdgraph repomap`: a pure personalized-PageRank ranking rendered to a
   token-budgeted `repomap.md`, reusing gdctx's budget config idiom.

| Goal | Metric | Target | Measurement |
|------|--------|--------|-------------|
| **G-B1** Tree-sitter symbol graph | symbol-edge precision/recall vs regex baseline on a labeled fixture | ≥ 0.9 precision on symbol/CALL edges; resolves function/class/method + CALL edges regex cannot | `fixtures/symbol-graph/` hand-labeled expected graph |
| **G-B2** Transitive `affected` | exactness of N-hop dependent set | `affected --depth N` == exact closure to depth N; `--depth 1` == today (byte-compatible) | `fixtures/transitive-closure/` known depth-1..k closures |

## Non-Goals (from `problem-statement.md` — MUST NOT appear as features)

- **NG-B1** No LSP / language-server runtime dependency, no semantic index server.
- **NG-B2** No vector/embedding-based code retrieval in this group (grep-over-RAG preserved; PageRank is graph-structural, not embedding-based).
- **NG-B3** No grammars beyond the project's languages + user-provided set.
- **NG-B4** No runtime/dynamic call-graph tracing — static edges only.

## Technical Foundation

- **Stack**: inherited — Bun runtime, TypeScript ESM, `node:fs/promises`, `Bun.Transpiler`, `Bun.spawn`. New optional lib: **`web-tree-sitter`** under `optionalDependencies` only (`C0-1`), lazy-imported inside the adapter (`C0-2`).
- **Architecture**: "Deterministic Core + Opt-In Capability Adapters" (`architecture.md` §0). B1 is a Capability (`gdgraph.treesitter`) with the four coordinated parts (`C0-3`): init flag, manifest entry, config toggle, `resolveCapability → Adapter | null` fallback. B2/B3 are pure service extensions requiring no capability.
- **Key MUST constraints**: `B-1` (regex fallback, exit 0), `B-2` (`web-tree-sitter` WASM, no native build), `B-3` (`--depth N` exact closure, `--depth 1` back-compat), `B-4` (pure PageRank, token-budgeted, reproducible), `B-5` (deterministic, git-diffable), `C0-7` (byte-identical fallback), `F-1`..`F-4` (fixtures as acceptance), `T-3`/`T-4` (availability-true/false tests, no-network sandbox).

## User Stories

### Epic B1 — Tree-sitter symbol graph (G-B1 / P-B1)

#### US-B101: Symbol-level nodes and CALL edges when tree-sitter is available
**As a** agent operator
**I want to** run `gdgraph build` and get function/class/method nodes plus import and CALL edges
**So that** I can see an accurate symbol-level dependency picture instead of a file-level import approximation (G-B1, solves P-B1)

**Acceptance Criteria:**
- [ ] With the `gdgraph.treesitter` capability enabled and grammars resolved, `build` writes `storage/symbols.jsonl` (function/class/method nodes) and `storage/calls.jsonl` (CALL + intra-file `defines` edges) in addition to the existing file-level artifacts.
- [ ] On `fixtures/symbol-graph/`, emitted symbol nodes and CALL/import edges match the hand-labeled expected set with ≥ 0.9 precision and the recall target fixed in `acceptance-criteria.md`.
- [ ] Symbol node ids are stable and deterministic (`<path>#<Container>.<name>` scheme); re-running `build` twice yields byte-identical symbol/call files.
- [ ] `web-tree-sitter` is loaded only via `await import()` inside `src/gdgraph/treesitter/adapter.ts`; no top-level import of it exists anywhere in `src/`.

**Technical Notes:** Constraint refs `B-1`,`B-2`,`B-5`,`C0-2`,`C0-14`,`F-1`,`F-4`. Architecture ref §5 Group B, §8 (B row), §2 capability seam.
**Priority:** P0

#### US-B102: Byte-identical regex fallback when tree-sitter is absent
**As a** agent operator on a default install
**I want to** run every `gdgraph` command with no `web-tree-sitter` and no grammars and get exactly today's behavior
**So that** the zero-dependency, zero-network floor is preserved (G-B1 preservation note, XP1/XP2)

**Acceptance Criteria:**
- [ ] With the capability disabled (or dep/grammar absent), `build` emits `nodes.jsonl`, `edges.jsonl`, `module-map.json`, `summary.md` **byte-for-byte identical** to the pre-block output on the same fixture repo.
- [ ] No `symbols.jsonl`/`calls.jsonl` are written when the capability is inactive; `web-tree-sitter` is never imported and no grammar file is read.
- [ ] Capability enabled but dep/grammar missing/checksum-failing ⇒ **one** stderr warning, deterministic regex path runs, exit code 0 (`C0-5`).
- [ ] A no-network sandbox run of `build`/`affected`/`repomap` opens no socket (`T-4`).

**Technical Notes:** Constraint refs `B-1`,`C0-4`,`C0-5`,`C0-7`,`C0-11`,`F-3`,`T-4`. Architecture ref §2.3 graceful-degradation contract, §3 Asset Resolver.
**Priority:** P0

#### US-B103: Grammar assets follow the XP3 delivery rules
**As a** agent operator opting into tree-sitter
**I want to** point at a local grammars directory or explicitly pull verified WASM grammars
**So that** no asset is bundled or silently downloaded (XP3)

**Acceptance Criteria:**
- [ ] Grammar WASM files are resolved through Block 0's Asset Resolver from one of: `treesitter.grammarsPath` config, `gd-metapro gdgraph assets pull <id>`, or the well-known cache (`A-1`).
- [ ] `assets pull` verifies sha256 against a committed `assets.lock.json` and refuses on mismatch (`A-2`); the resolver re-verifies on every load and returns unavailable ⇒ fallback on mismatch/missing (`A-3`).
- [ ] No default command reaches the download path; there is no `postinstall` fetch (`A-6`,`A-7`,`C0-12`).

**Technical Notes:** Constraint refs `A-1`..`A-7`,`C0-12`. Architecture ref §3 Asset Delivery.
**Priority:** P0

### Epic B2 — Transitive `affected` (G-B2 / P-B2)

#### US-B201: N-hop blast radius with configurable depth
**As a** agent operator scoping a change
**I want to** run `gdgraph affected <file> --depth N` and get the transitive dependent closure to depth N
**So that** I see the true blast radius, not just direct dependents (G-B2, solves P-B2)

**Acceptance Criteria:**
- [ ] `affected <file> --depth N` returns the **exact** transitive closure of dependents to depth N on `fixtures/transitive-closure/` for N = 1..k.
- [ ] `--depth` defaults such that omitting it and `--depth 1` both reproduce **today's exact stdout** (Dependencies + Dependents sections, sorted) — back-compat verified byte-for-byte (`B-3`).
- [ ] Cycles in the graph terminate (visited-set) and never loop; output is deterministic and sorted.
- [ ] Algorithm is pure (no dep, no I/O beyond loading the graph), operating on the in-memory graph from `loadGraph`.

**Technical Notes:** Constraint refs `B-3`,`B-5`,`B-9`,`F-4`. Architecture ref §5 Group B ("pure BFS/DFS over the in-memory graph").
**Priority:** P0

#### US-B202: Ranked blast-radius output
**As a** external coding agent selecting tests/reviews
**I want to** get affected dependents ranked by proximity and fan-in
**So that** I can prioritize the highest-impact nodes first (G-B2)

**Acceptance Criteria:**
- [ ] `affected --depth N --ranked` (and/or `--json`) emits each dependent with its hop distance and a blast-radius rank; ties broken deterministically by path.
- [ ] Ranking is a pure function of the graph (hop distance ascending, then fan-in descending, then path); reproducible across runs.
- [ ] The default (unflagged) text output remains byte-identical to today (`--ranked`/`--json` are additive opt-ins).

**Technical Notes:** Constraint refs `B-3`,`B-5`. Architecture ref §5 Group B (ranked blast-radius). Works at symbol granularity once B1 is active, file granularity otherwise.
**Priority:** P1

### Epic B3 — Ranked, token-budgeted repo map (G-B3 / P-B3)

#### US-B301: Personalized-PageRank `repomap.md`
**As a** external coding agent needing repo context
**I want to** run `gdgraph repomap` and get a compact, centrality-ranked map of the repo
**So that** I get high-signal context injection instead of dumping the tree (G-B3, solves P-B3)

**Acceptance Criteria:**
- [ ] `repomap` computes personalized PageRank with edge weights over the graph (symbol-level when B1 active, file-level otherwise) using a fixed iteration count / convergence tolerance, and renders ranked entries (path + top symbols + signatures) to `.metaproject/data/gdgraph/artifacts/repomap.md`.
- [ ] Ordering is deterministic and reproducible: re-running `repomap` twice yields a byte-identical `repomap.md` (empty diff).
- [ ] Top-ranked entries match the expected centrality ordering on `fixtures/repomap/`.
- [ ] No runtime dependency and no network is used (`B-4`, pure algorithm; NG-B2 preserved).

**Technical Notes:** Constraint refs `B-4`,`B-5`,`B-7`,`F-4`. Architecture ref §5 Group B (Aider-style personalized PageRank).
**Priority:** P0

#### US-B302: Token-budget enforcement reusing gdctx budget-awareness
**As a** external coding agent with a fixed context window
**I want to** the repo map to never exceed a configured token budget
**So that** it always fits the injection budget (G-B3)

**Acceptance Criteria:**
- [ ] A `repomap.tokenBudget` config field (deep-merged over a default, `C0-8`, mirroring `gdctx.config.json` budget fields) caps output; entries are included in PageRank-rank order until the budget is reached, then truncated with a stable "… N entries omitted …" marker.
- [ ] The emitted `repomap.md` token estimate (documented estimator, e.g. chars/4, reusing the gdctx byte-budget idiom) is ≤ the configured budget on `fixtures/repomap/`.
- [ ] `--budget <n>` CLI override takes precedence over config; both are deterministic.

**Technical Notes:** Constraint refs `B-4`,`C0-8`,`F-4`. Existing pattern: `src/commands/ctx.ts` `CtxConfig`/`loadConfig` deep-merge + head/tail truncation. Architecture ref §5 Group B ("token-budgeted via existing gdctx token-awareness").
**Priority:** P0

#### US-B303: Personalization seed for change-scoped maps
**As a** agent operator reviewing a change
**I want to** bias the repo map toward changed files
**So that** the map is most detailed where I am working (G-B3)

**Acceptance Criteria:**
- [ ] `repomap --seed <path...>` (or `--changed`) sets the PageRank personalization vector to the given nodes; with no seed the personalization is uniform (deterministic default).
- [ ] Seeded and unseeded runs are each independently reproducible.

**Technical Notes:** Constraint refs `B-4`,`B-5`. Architecture ref §5 Group B (personalized PageRank).
**Priority:** P2

## Data Model (High-Level)

Additive superset of `src/gdgraph/types.ts`; file-level types unchanged (see `specification.md` §Schema). New symbol nodes (`function`/`class`/`method`/`interface`), new edges (`calls`/`defines`), written to new `storage/symbols.jsonl` + `storage/calls.jsonl` only when the tree-sitter capability is active.

## API Surface (High-Level)

- `gd-metapro gdgraph build` — unchanged default output; writes symbol layer additively when capability active.
- `gd-metapro gdgraph affected <file> [--depth N] [--ranked] [--json]` — N-hop.
- `gd-metapro gdgraph repomap [--budget N] [--seed <path...>|--changed]` — new.
- `gd-metapro gdgraph assets pull|list|verify <id>` — grammar assets (Block 0 subcommand surface).

## Testing Requirements (from `tech-bestpractices.md` §8)

- Unit (~70%): pure BFS/DFS closure, PageRank, token-budget truncation, symbol-id derivation, config deep-merge, regex-fallback path.
- Integration (~25%): capability availability-true (grammars stubbed) vs availability-false (`T-3`); config/manifest wiring; asset resolution.
- E2E/CI (~5%): fixture-corpus assertions (`F-1`); no-network sandbox (`T-4`); byte-identical fallback snapshot (`F-3`,`C0-7`).

## Out of Scope

All of NG-B1..NG-B4 (LSP/server, embedding retrieval, non-project grammars, dynamic tracing). MCP exposure of these actions belongs to Block A. Memory/embedding retrieval belongs to Block C.
