## 1. Problem

gd-metapro's whole 2026 extension effort has one tension: **add model / embedding / network /
precision features WITHOUT breaking the zero-dependency, zero-network deterministic default.**
The proven answer already lives in the codebase — `src/security/` runs deterministic regex
detectors by default and gates optional model backends behind `config.backends.*.enabled`, with
every seam (`guardOutput`, `redactRaw`) returning a graceful no-op when disabled and **never
throwing**. But that idiom is **module-local to `security`**. If each of Blocks A–E re-invents its
own opt-in wiring, asset handling, and fallback contract, the zero-dep floor will erode block by
block (the discovery brief flagged "decide the first-runtime-dep policy once, centrally" as the
gating risk).

There is today **no shared mechanism** for: (a) declaring an optional capability across all four
coordinated parts uniformly; (b) loading an optional runtime dependency lazily without polluting
`package.json` `dependencies`; (c) resolving user-provided / opt-in-downloaded assets with
checksum verification and no silent network; or (d) proving a capability's quality against a
committed labeled fixture corpus. Block 0 builds exactly that substrate — and nothing else.

## 3. Non-Goals

Traceable to `problem-statement.md` non-goals and this block's scope boundary:

- **NG0-1**: Ship **no end-user feature**. Block 0 delivers only the substrate; MCP, tree-sitter,
  embeddings, hotspot signals, etc. belong to A–E.
- **NG0-2**: No `postinstall`/auto-download of any asset or dependency (`C0-12`, XP3).
- **NG0-3**: No bundling of models/grammars/embeddings into the package (arch §3, `A-5`, XP3).
- **NG0-4**: No mandatory runtime dependency, model, embedding, or network call on any default code
  path (`C0-10`, XP1).
- **NG0-5**: Not a monorepo/per-feature package split (arch §0 rejected alternative) — one package,
  `optionalDependencies` + lazy import.
- **NG0-6**: No auth/identity, no hosted service, no HTTP surface introduced by this block (those
  are A's concern and separately opt-in).

## 4. Technical Foundation

- **Stack (inherited, not re-chosen)**: Bun + TypeScript, ESM, layered `cli → command → service →
  lib`, zero-dep / zero-network deterministic core.
- **Architecture**: "Deterministic Core + Opt-In Capability Adapters" (arch §0). The seam lives
  **beside its service** (`src/<module>/<cap>/adapter.ts`); the shared resolver + harness live in a
  new shared location. `resolveCapability(cwd, name) → CapabilityAdapter | null` centralizes lazy
  import + graceful degrade.
- **Key constraints (MUST)**: `C0-1`..`C0-14`, `A-1`..`A-7`, `F-1`..`F-4`, `T-1`..`T-4` from
  `tech-bestpractices.md`.

## 5. User / Agent Stories

### US-001: One uniform way to declare an opt-in capability
**As a** gd-metapro maintainer building Block A–E
**I want to** declare an optional capability through one API that wires the init flag, the
`metaproject.json` manifest entry, the config toggle, and the `resolveCapability()` fallback
**So that** every block instantiates the opt-in mechanism identically instead of re-inventing it
(goal G0-1; solves the "decide dep policy once, centrally" risk in problem-statement §Assumptions).

**Acceptance Criteria:**
- [ ] A capability is registered by naming its four parts once; `resolveCapability(cwd, id)`
      returns an `Adapter` when all parts are satisfied and `null` otherwise.
- [ ] Disabling the capability in the manifest yields `null` with **no** dep loaded and **no**
      asset touched.
- [ ] A reference capability is wired end-to-end using only this API (no ad-hoc wiring).

**Technical Notes:** Constraint ref `C0-3`, `C0-9`; Architecture ref §2.1, §2.2.
**Priority:** P0

### US-002: The default install stays dependency-free
**As an** agent operator cloning gd-metapro
**I want to** run every default command with an empty `dependencies` block and no optionals
installed
**So that** the zero-dep / zero-network floor holds byte-identically to today (goal G0-2; XP1).

**Acceptance Criteria:**
- [ ] `package.json` `dependencies` is empty; new libs appear only under `optionalDependencies`.
- [ ] No optional dependency is imported at module top-level; it is loaded only via
      `await import(...)` inside an adapter, wrapped in try/catch.
- [ ] A default command runs successfully with the optionals directory stripped.

**Technical Notes:** Constraint ref `C0-1`, `C0-2`, `C0-13`; Architecture ref §2.2.
**Priority:** P0

### US-003: Missing dep/asset degrades gracefully, never crashes
**As an** agent operator who enabled a capability but has not installed its dep or asset
**I want** the command to warn once and fall back to the deterministic path with exit 0
**So that** an incomplete opt-in never breaks my workflow (goal G0-3; XP2).

**Acceptance Criteria:**
- [ ] Capability enabled but dep not installed / asset missing / checksum fails ⇒ warn once to
      stderr, run the deterministic fallback, **exit 0**.
- [ ] The warning is emitted **once** per command invocation (warn-once), not per call site.
- [ ] The deterministic fallback output is byte-identical to the capability-off output.

**Technical Notes:** Constraint ref `C0-5`, `C0-6`, `C0-7`, `F-3`; Architecture ref §2.3.
**Priority:** P0

### US-004: Adapters never throw out to a caller
**As a** service calling a capability adapter
**I want** any adapter or runtime error to be caught and degraded to the deterministic result
**So that** an opt-in ceiling can never break a deterministic seam (goal G0-4; mirrors
`analyze()`/`guardOutput` try/catch → allow).

**Acceptance Criteria:**
- [ ] An adapter that throws at `isAvailable()` or `run()` is caught; the caller receives the
      deterministic result.
- [ ] No adapter error propagates as an uncaught exception in any test.

**Technical Notes:** Constraint ref `C0-11`; Architecture ref §2.3(3).
**Priority:** P0

### US-005: Assets are user-provided or explicitly pulled, checksum-verified
**As an** agent operator opting into a model/grammar asset
**I want** the asset to come from a config path, an explicit `assets pull`, or a user cache — and
be sha256-verified on every load
**So that** nothing is bundled or silently downloaded and provenance is diffable (goal G0-5; XP3/XP4).

**Acceptance Criteria:**
- [ ] `resolveAsset(cfg, id)` resolves T1 user-path / T2 pulled / T3 cache; verifies sha256 on
      every load; returns `null` when missing or unverified.
- [ ] `gd-metapro <module> assets pull <id>` is the **only** path that touches the network; it
      verifies the download against `assets.lock.json` and refuses on mismatch.
- [ ] `assets.lock.json` (id/version/url/sha256/size) is git-committed.
- [ ] No default command reaches the resolver's download path (no-network sandbox passes).

**Technical Notes:** Constraint ref `A-1`..`A-7`, `C0-12`; Architecture ref §3.
**Priority:** P0

### US-006: Prove capability quality against a committed fixture corpus
**As a** maintainer shipping an opt-in capability
**I want** a reusable harness that runs a labeled fixture corpus and reports a per-detector
false-negative rate with a regression threshold
**So that** each block proves quality against data, not prose, and CI catches silent regressions
(goal G0-6; F-1..F-4, generalizes E6).

**Acceptance Criteria:**
- [ ] The harness loads a `fixtures/<corpus>/` directory, runs the block's detector/selector, and
      emits an FN-rate (and precision/recall) report.
- [ ] The report is deterministic and re-runnable (re-run diff empty).
- [ ] The harness exposes a CI-gating entry that fails when FN rate regresses beyond a threshold.
- [ ] A block PRD can name its corpus as the acceptance gate; the harness accepts it without
      per-block code.

**Technical Notes:** Constraint ref `F-1`, `F-2`, `F-4`, `E-8`; Architecture ref §6.
**Priority:** P0

### US-007: `init` and `update` register capabilities uniformly
**As an** agent operator running `gd-metapro init`/`update`
**I want** each opt-in capability to be offered via `--<cap>` / `--no-<cap>` (ceilings default OFF)
and written into `metaproject.json` + the module config
**So that** enabling a ceiling is one consistent gesture across all blocks (goal G0-1; XP2).

**Acceptance Criteria:**
- [ ] `init` accepts `--<cap>` / `--no-<cap>`; ceilings default OFF (mirrors `--no-security`).
- [ ] The chosen capabilities are written to `modules.<m>.capabilities[]` (and/or a `capabilities`
      block) and to the module's `*.config.json`, deep-merged over defaults.
- [ ] `update` reconciles a newly-added capability into an existing manifest without disabling
      already-enabled modules.
- [ ] A missing manifest = capability off (mirrors `isSecurityEnabled`).

**Technical Notes:** Constraint ref `C0-3`, `C0-8`, `C0-9`; Architecture ref §2.1.
**Priority:** P0

## 6. XP1–XP5 Requirements Delivered by This Block

This block is where each cross-cutting policy becomes an **enforced mechanism** the other blocks
inherit:

| XP | Policy | What Block 0 delivers |
|----|--------|-----------------------|
| **XP1** | Deterministic core is zero-dep / zero-network | Empty `dependencies` + `optionalDependencies` policy + lazy `await import()` (US-002); no-network sandbox test (US-005) |
| **XP2** | Opt-in only, graceful no-op / fallback | The four-part opt-in wiring + `resolveCapability → Adapter \| null` + warn-once/exit-0 fallback (US-001, US-003) |
| **XP3** | Assets never bundled, never auto-downloaded | Asset Resolver three-tier + `assets pull` as the sole network path + `assets.lock.json` (US-005) |
| **XP4** | Determinism & leak-safety preserved | Checksum verification on every load + committed lockfile + deterministic fallback + reproducible harness (US-005, US-006) |
| **XP5** | MCP transport is stdio-first | Not delivered here (transport is Group A); Block 0 only guarantees the seam an MCP capability plugs into. Noted for traceability. |

## 7. Testing Requirements

Per `tech-bestpractices.md` §8 (pyramid ~70/25/5 for this package):
- **Unit (~70%)**: seam resolution (available/unavailable), fallback path, adapter-never-throws,
  config deep-merge + malformed-JSON fallback, checksum verify/refuse, harness FN-rate math.
- **Integration (~25%)**: `init`/`update` capability wiring into `metaproject.json` + config;
  `assets pull` → verify → resolve round-trip.
- **E2E/CI (~5%)**: no-network sandbox over default commands; harness CI-gate self-test.
- Each opt-in capability MUST have both an availability-true and an availability-false test
  (`T-3`); the reference capability shipped with Block 0 demonstrates the pattern.
